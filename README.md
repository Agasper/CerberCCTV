# Cerber CCTV

Домашнее видеонаблюдение из двух частей: **агент** на Raspberry Pi читает
RTSP-поток IP-камеры (Hikvision и любой другой с RTSP), детектит движение и
собирает клипы, а **админка** на внешнем сервере принимает эти клипы, кладёт их
в S3-совместимое хранилище и показывает в веб-интерфейсе — список записей с
плеером, живая трансляция, настройки. Всё запускается в Docker.

Ключевое свойство: **дома не нужен статический IP и проброс портов**. Агент
ходит к админке только исходящими HTTPS/WSS-соединениями, поэтому работает за
любым NAT. Видео агент загружает в админку, а она сама пишет в S3 — ключи от
хранилища никогда не попадают на Raspberry Pi.

```
┌──────── дом (NAT, нет статики) ────────┐   ┌──── DO App Platform ─────┐
│ Камера ──RTSP──> Агент (Docker)        │   │  Админка FastAPI (:8080, │
│   основной поток -> ffmpeg ring buffer │   │  TLS от платформы)       │
│   субпоток -> OpenCV motion            │   │    │            │        │
│   клип ──HTTPS (chunked)────────────────────>  upload API   S3       │
│   heartbeat/конфиг/команды ──WSS────────────>  agent API             │
│   live fMP4 ──WSS───────────────────────────>  WS-relay ──MSE──> браузер
└────────────────────────────────────────┘   │        Postgres (внешний)│
                                             └──────────────────────────┘
```

## Как это работает

**Запись без перекодирования.** ffmpeg на агенте непрерывно копирует основной
поток камеры в кольцевой буфер из сегментов по 2 секунды (`-c copy`, CPU почти
не тратится). Буфер живёт в tmpfs — SD-карта Raspberry Pi не изнашивается.

**Детекция движения.** OpenCV читает низкоразрешённый субпоток камеры
(вычитание фона MOG2 → контуры → площадь движения в % кадра). Чувствительность,
минимальная площадь, пре/пост-ролл, пауза между событиями и максимальная длина
клипа настраиваются в админке и применяются агентом на лету, без рестарта.
Там же рисуются **зоны детекции** — полигоны поверх снапшота камеры; движение
вне зон игнорируется (порог площади считается от площади зон).

**Событие движения.** При срабатывании агент забирает из буфера сегменты
пре-ролла, ждёт окончания движения (плюс пост-ролл), склеивает всё в mp4
(faststart, играется в браузере) и кладёт в локальную очередь (outbox).

**Доставка.** Клип грузится в админку чанками по 8 МиБ с `Content-Range` —
загрузка возобновляется с места обрыва. Каждый чанк админка сразу отправляет
part'ом в S3 multipart upload, не полагаясь на свой диск (в App Platform он
эфемерный). Пока админка недоступна, клипы копятся в outbox и досылаются позже.

**Live.** По кнопке в админке агент получает команду по WebSocket, запускает
ffmpeg (снова stream copy) и пушит fMP4-фрагменты по WSS в админку, которая
раздаёт их зрителям; в браузере — MSE-плеер. Задержка 1–3 секунды. Трансляция
гаснет сама, когда все зрители ушли (и по таймауту на всякий случай).

**Отказоустойчивость.** Обе стороны переживают рестарты друг друга: агент
кэширует конфиг локально и копит клипы в outbox, админка хранит состояние
multipart-загрузок в Postgres. ffmpeg и детектор перезапускаются при обрыве
потока и при «застывании» камеры.

## Структура репозитория

```
common/   pydantic-модели API-контракта агент<->админка
agent/    агент для Raspberry Pi (захват, движение, outbox, live)
admin/    админка FastAPI (UI, приём клипов, S3, live-relay, ретенция)
.do/      App Spec для деплоя в DigitalOcean App Platform
docker-compose.dev.yml  локальная e2e-среда с фейковой камерой и MinIO
```

## Быстрый старт: локальная среда (без железа и облака)

Нужен только Docker. Поднимается всё сразу: Postgres, MinIO (локальный S3),
фейковая RTSP-камера (MediaMTX + ffmpeg с движущейся тест-картинкой), админка
и агент:

```bash
docker compose -f docker-compose.dev.yml up -d --build
```

Через минуту-две:

- админка: http://localhost:8090 — логин `admin`, пароль `admin123`;
- на дашборде появится снапшот «камеры» и статус агента;
- тест-картинка движется непрерывно, поэтому события создаются одно за другим
  (каждое длится до `max_clip_s`, по умолчанию 120 с) — записи появляются на
  странице «События» и проигрываются из MinIO;
- консоль MinIO: http://localhost:9001 (`cerber` / `cerber-secret`).

## Деплой в прод

### 1. Админка — DigitalOcean App Platform

Предполагается, что Postgres у вас уже есть (managed-БД DO или свой), и есть
S3-совместимое хранилище (DO Spaces, Backblaze, MinIO на VPS и т.п.).

1. Форкните/запушьте репозиторий на GitHub (спека ссылается на
   `Agasper/CerberCCTV`, поправьте при необходимости).
2. Создайте приложение из спеки:
   ```bash
   doctl apps create --spec .do/app.yaml
   ```
   или через UI (Create App → импорт репозитория). Платформа соберёт
   `admin/Dockerfile` и выдаст домен вида `https://<app>.ondigitalocean.app`
   с TLS.
3. Задайте переменные окружения (секреты):
   - `DATABASE_URL` — строка подключения к Postgres (для managed-БД DO —
     с `?sslmode=require`; параметр обрабатывается корректно);
   - `SECRET_KEY` — любая длинная случайная строка (подпись cookie);
   - `ADMIN_PASSWORD` — пароль первого входа (используется один раз при
     пустой БД);
   - `AGENT_TOKEN` — можно не задавать: тогда после входа выпустите токен на
     странице «Настройки».
4. Миграции накатываются автоматически при старте контейнера (alembic в
   entrypoint).
5. Зайдите в админку → «Настройки» → заполните раздел S3 (endpoint, bucket,
   ключи; для DO Spaces endpoint вида `https://fra1.digitaloceanspaces.com`)
   и нажмите «Проверить подключение».

### 2. Агент — Raspberry Pi

Нужен Raspberry Pi с **64-битной** ОС (для колёс opencv под aarch64) и Docker.

```bash
git clone https://github.com/Agasper/CerberCCTV.git
cd CerberCCTV/agent
cp .env.example .env        # вписать ADMIN_URL и AGENT_TOKEN
docker compose up -d --build
```

Дальше всё настраивается из админки, раздел «Камера»:

- основной поток: `rtsp://admin:ПАРОЛЬ@192.168.0.10:554/Streaming/Channels/101`
- субпоток: `rtsp://admin:ПАРОЛЬ@192.168.0.10:554/Streaming/Channels/102`

Учётка камеры хранится только в БД админки; в репозитории и на диске Pi её нет.

### 3. Настройка камеры (Hikvision)

- Включите субпоток (Substream) с небольшим разрешением, например
  640×360 @ 5–10 fps — его анализирует детектор, чем меньше, тем легче Pi.
- Кодек видео — **H.264**. H.265 сохранится и загрузится, но live-просмотр и
  воспроизведение в браузере зависят от поддержки HEVC конкретным
  браузером/железом; с H.264 работает везде.
- I-frame interval (GOP) основного потока разумно держать 1–2 секунды: от него
  зависят точность нарезки сегментов и задержка live.

## Безопасность

- Админка — сессионная авторизация (bcrypt, подписанные cookie), смена пароля
  в настройках. Агент — Bearer-токен, в БД хранится только его sha256-хэш,
  перевыпуск в один клик.
- App Platform терминирует TLS: агент и браузер ходят по HTTPS/WSS.
- Ключи S3 живут только в БД админки. Записи отдаются короткоживущими
  presigned-ссылками.

## Ограничения и заметки

- Один агент = одна камера. Схема БД заложена под несколько агентов, но UI
  и live-relay сейчас рассчитаны на одного.
- Live на iPhone: Safari поддерживает MSE начиная с iOS 17.1
  (ManagedMediaSource); на более старых iOS живой просмотр работать не будет
  (записи — будут). Не проверялось на реальном устройстве.
- Звук: в клипах сохраняется, если камера отдаёт AAC; в live звук отключён
  сознательно (упрощение и экономия трафика).
- Точный лимит размера тела запроса у ингресса App Platform документацией не
  гарантирован — потому загрузка и идёт чанками по 8 МиБ.
- Проверено end-to-end в локальной dev-среде (фейковая камера + MinIO +
  Postgres в Docker), а также с реальной камерой Hikvision (H.264
  1080p/25fps, оба потока). Деплой в App Platform описан по документации,
  но на живой инфраструктуре автором не прогонялся — если что-то не
  сойдётся, начните разбор с логов `docker logs` агента и Runtime Logs
  приложения в DO.
